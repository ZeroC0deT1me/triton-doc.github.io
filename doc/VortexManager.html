echelon.module.vortex

VortexManager

Package echelon.module.vortex Type public final class · singleton Owns VortexLoader

High-level coordinator for the Vortex class-loading system. Manages the lifecycle of a VortexLoader, the on-disk directory structure, the vortex_manifest.json registry, staging, snapshot/rollback, and the programmatic deploy API. Initialize it once at app startup — all subsequent module loads automatically benefit from any absorbed or included classes.

Quick Start

public static void main(String[] args) {
    // 1. Initialize Vortex FIRST — before any app classes are touched
    VortexManager vortex = VortexManager.initialize();

    // 2. Optional: observe what gets absorbed
    vortex.getLoader().setListener((className, event) -> {
        switch (event) {
            case ABSORBED -> System.out.println("Absorbed: " + className);
            case REJECTED_LOCKED -> System.out.println("Sealed: " + className);
            case FAILED  -> System.err.println("Failed:  " + className);
        }
    });

    // 3. Launch app normally — Vortex is transparent from here on
    EchelonDesktop.launch(args);
}

initialize()

Two overloads — both use double-checked locking and are safe to call any number of times:

OverloadResolves root from
VortexManager.initialize()DirectoryConfig.shared().getMain() + "vortex-module" — standard app layout.
VortexManager.initialize(Path vortexRoot)Explicit path — useful for tests or non-standard deployments.

On first call, setup() runs: creates system/, staging/, and snapshots/ directories if they don't exist, loads the manifest, sets the current thread's context classloader to the VortexLoader, and logs a startup summary.

Use VortexManager.getInstance() to retrieve the singleton after initialization — returns null if not yet initialized.

Timing Requirement

Must be first VortexManager.initialize() must be called before any class you intend to override is referenced. Once the JVM loads and caches a class, it will not reload it for the lifetime of the process — that is why a restart is required for new overrides to take effect.
Thread context classloader setup() sets the initializing thread's context classloader to the VortexLoader. If you spawn raw threads later, set their context classloader explicitly: thread.setContextClassLoader(VortexManager.getInstance().getLoader()).

deploySystemOverride()

Writes compiled .class bytes into system/ under the correct package path, records the entry in the manifest, and persists vortex_manifest.json. The override takes effect on next restart.

// Override an existing framework class
byte[] newBytes = Files.readAllBytes(patchedClassFile);
vortex.deploySystemOverride("echelon.ui.WindowManager", newBytes, "1.5.0");
vortex.requestRestart(() -> EchelonDesktop.restart());

deployInclude()

Same as deploySystemOverride() but writes to the vortex root (not system/). For classes that do not exist in the JAR. If the class already exists in the JAR, VortexLoader will reject it at load time with REJECTED_INCLUDE_CONFLICT.

deployJar()

Copies a .jar file into the appropriate directory (system/ for override mode, vortex root for include mode), registers it in the manifest under the key jar:filename.jar, then calls VortexLoader.rescanJars() so the new JAR is indexed immediately. Already-loaded classes still require a restart.

// Deploy a hotfix JAR — overrides existing classes
vortex.deployJar(
    Path.of("hotfix-1.5.jar"),
    VortexEntry.Type.SYSTEM,
    "1.5.0"
);

deployFromFile()

Reads bytes from a Path on disk and delegates to the appropriate deploy method. Convenience wrapper for when you already have the .class file sitting on disk rather than in memory.

absorbFromStaging()

Processes all .class files in staging/ and moves them to the correct live directories, mirroring the staging layout:

Staging pathDeployed toType
staging/system/echelon/ui/X.classsystem/echelon/ui/X.classSYSTEM override
staging/echelon/newpkg/Y.classechelon/newpkg/Y.class (root)INCLUDE

Returns the count of classes deployed. Empty staging directories are cleaned up automatically. Manifest is saved after the batch.

// Workflow: snapshot → stage → absorb → restart
String snap = vortex.createSnapshot("pre-2.0");
// ... drop files into staging/ ...
int count = vortex.absorbFromStaging();
System.out.println("Staged " + count + " class(es)");
vortex.requestRestart(() -> EchelonDesktop.restart());

remove() & purgeAll()

remove(className) looks up the manifest entry, deletes the file on disk, removes the entry, saves the manifest, and — if it was a JAR — calls rescanJars(). Empty package directories are cleaned up after loose-class removals.

purgeAll() calls remove() for every manifest entry in sequence. Returns the count of successfully removed entries. After a purge, the next restart will use the original JAR for all classes.

Snapshots & Rollback

Snapshots capture the current state of all overrides and includes so you can roll back if an update breaks something.

MethodDescription
createSnapshot(String label)Copies system/, root includes, and vortex_manifest.json into snapshots/label_timestamp/. Returns the snapshot ID.
restoreSnapshot(String id)Clears current overrides/includes and restores from the snapshot. Requires restart to take effect.
deleteSnapshot(String id)Removes the snapshot directory permanently.
listSnapshots()Returns a sorted list of snapshot IDs.
 
// Snapshot → deploy → rollback if needed
String snap = vortex.createSnapshot("pre-hotfix");
vortex.deployJar(hotfixPath, VortexEntry.Type.SYSTEM, "1.5.0");

// Something broke?
vortex.restoreSnapshot(snap);
vortex.requestRestart(() -> EchelonDesktop.restart());

Restart

requestRestart(Runnable) logs the restart intent and immediately calls the provided callback — it does not itself restart the process. The callback is typically EchelonDesktop.restart() or an equivalent app-specific restart routine.

isRestartNeeded() returns true if any .class files remain in staging/ — a signal that staged changes are waiting to be absorbed and activated.

VortexEntry

Fields

The manifest record for a single deployed class or JAR. Serialized to / deserialized from vortex_manifest.json using a hand-rolled parser (no external JSON library).

FieldTypeDescription
classNameStringFully-qualified class name, or jar:filename.jar for JAR entries.
versionStringVersion label provided at deploy time. "staged" for staging-absorbed classes.
typeTypeSYSTEM (override) or INCLUDE (new class).
deployedAtStringISO-8601 instant string of when the entry was deployed.
filePathStringPath relative to vortexRoot — used by remove() to locate the file.
sizeBytesintFile size in bytes at deploy time.

ValueMeaning
